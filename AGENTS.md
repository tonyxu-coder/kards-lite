# AGENTS.md

本文件给之后参与这个仓库的人或 AI 助手使用。

目标很简单：继续把这个小游戏做下去，但不要把项目改乱。

## 1. 项目定位

- 这是一个纯前端小游戏项目。
- 当前主要文件是 `index.html`、`app.js`、`styles.css`。
- 项目已经部署到 GitHub Pages：
  - 仓库地址：`https://github.com/tonyxu-coder/kards-lite`
  - 访问地址：`https://tonyxu-coder.github.io/kards-lite`
- 默认假设：项目应该继续保持“打开就能运行、方便分享”的方向。

## 2. 开发总原则

- 优先做小步修改，不要一次性大改全部逻辑。
- 先读懂现有代码，再动手改。
- 没有明确需求时，不要随便重构整个项目。
- 保持代码简单，适合当前作者继续看懂和维护。
- 如果一个实现方式更高级，但更难懂，优先选更容易维护的方案。

## 3. 技术边界

- 默认继续使用原生 `HTML + CSS + JavaScript`。
- 不要随便引入打包器、后端服务、数据库、复杂框架。
- 不要让项目依赖 Node.js 才能运行，除非用户明确要求升级架构。
- 所有资源路径尽量使用相对路径，保证 GitHub Pages 可直接部署。

## 4. 修改代码时要注意

- 改界面时，同时检查电脑和手机显示是否还能正常使用。
- 改 `app.js` 时，注意不要破坏已有卡组、阵营、地形、关键词之间的对应关系。
- 如果新增卡牌、阵营或规则，命名要统一，避免中英文混乱。
- 如果修改了玩法规则，相关说明文字也要一起更新。
- 如果修改会影响链接直达功能，必须检查 URL 参数还能不能用。
- 不要删除已有内容，除非确认它已经废弃并且不会影响现有功能。

## 5. 玩法设计规则

- 新内容要优先保证“能玩懂”，再追求“花哨”。
- 不要让某一张卡或某个阵营明显强到没有对手。
- 新关键词要尽量短、好理解、能在界面里讲清楚。
- 每个阵营最好保留自己的特色，不要做成只是换名字。
- 这是战争题材游戏，但文案应避免过度美化战争，重点放在策略玩法。

## 6. 文案规则

- 默认使用简体中文。
- 文案尽量短，适合玩家快速看懂。
- 名称、描述、提示语的风格要统一。
- 面向当前作者时，说明应直接、清楚，不要写得太学术。

## 7. 测试与检查

每次改完后，至少检查下面这些内容：

1. 首页能正常打开。
2. 主要按钮能点击。
3. 选择卡组或模式后不会直接报错。
4. 战斗流程至少能完整打一局。
5. 手机上排版没有明显错位。
6. GitHub Pages 路径下资源能正常加载。

本地测试可用：

```bash
python3 -m http.server 8766
```

然后打开：

```text
http://127.0.0.1:8766
```

## 8. 文档同步规则

- 如果改了部署地址、仓库地址、启动方式，要同步更新 `README.md`。
- 如果新增重要规则，最好同步更新本文件。
- 如果加入了新的开发约定，也写进本文件，避免以后忘记。

## 9. 给 AI 助手的要求

- 先看现有文件，再提出改动。
- 先做最小可行修改，不要默认“大重构”。
- 回答尽量清楚，适合初中生继续接手。
- 提建议时，优先给出简单方案和原因。
- 如果发现风险较大的改动，要先明确说明风险点。

## 10. 不要做的事

- 不要私自把项目改成复杂工程化项目。
- 不要随便引入一堆外部依赖。
- 不要未经说明就改掉核心玩法。
- 不要把能直接运行的静态站点改成必须配置很多环境才能启动。
- 不要写只有你自己看得懂的代码或说明。
